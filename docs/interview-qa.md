# Shopkeeper Agent — 面试准备文档

> 针对简历中这个项目的所有可能提问，按技术领域分类，包含面试官视角和回答策略。

---

## 一、项目概览类

### Q1：简单介绍一下这个项目

**面试官意图**：考察你能否在 1-2 分钟内说清楚项目的核心价值。

**推荐回答（30 秒版）：**

> 这个项目解决的是电商运营和内部文档查询的效率问题。运营每天要查 GMV、销量、区域分布这些数据，传统流程是提 BI 工单等排期。系统把 NL2SQL 和 RAG 两条线串起来——运营用自然语言就能拿到结果，响应时间从天级降到秒级。

### Q2：这个项目是你一个人做的吗？你在其中负责什么？

**面试官意图**：想确认你的实际参与程度。

**回答策略**：个人项目直接说全部由我独立完成。如果是跟随教程的，诚实说明但强调独立改进的部分。

## 二、NL2SQL 深度类

### Q3：LangGraph 相比直接调 LLM 有什么好处？

> 可观测和可控。拆成 11 个节点后，每个节点独立输出进度，前端实时看到执行步骤。某步失败可以只重试该步。条件路由：SQL 校验失败自动走修正分支，最多重试 2 次，修正后回到校验节点重新验证。

**追问：为什么不用 Chain 而用 StateGraph？**
> Chain 是线性的。我们的场景需要并行召回（字段/指标/取值三条路），StateGraph 天然支持 fan-out/fan-in。

### Q4：SQL 校验和修正怎么做？

> 两层。语法校验用 MySQL EXPLAIN，报错就传给修正节点重新生成，最多重试 2 次。安全校验用 sqlglot 解析 AST 后全树遍历，拒绝任何非 SELECT 的节点。

### Q5：三路召回具体是什么？

> 字段向量召回、指标向量召回、字段取值全文检索。用户说"业绩"但库里叫"order_amount"，需要向量语义匹配。字段取值单独走 ES 是因为它要求精确匹配。

### Q6：怎么防止 LLM 编造表名和字段名？

> 召回阶段把检索到的真实字段信息传给 LLM 限制范围。校验阶段 EXPLAIN 会直接报错，修正节点根据报错重新生成。

## 三、RAG 深度类

### Q7：为什么 Qdrant + ES 两条路？

> 互补。Qdrant 向量检索擅长语义匹配，ES BM25 擅长精确匹配。合并时对两边分数做归一化（BM25 用 sigmoid 映射到 [0,1]），按权重加权排序截断 top-K。

### Q8：文档切分策略？

> 句子级切分，最大 256 token。存两层结构：子块用于检索，父块用于生成回答。检索命中最相关的子块，取对应父块传 LLM。

## 四、工程与架构类

### Q9：为什么 FastAPI？

> 异步支持。三个 Agent 都需要 SSE 流式输出，FastAPI 原生支持 async。Django 太重——只有 13 个路由。

### Q10：测试覆盖？

> 目前是短板。有端到端手工测试脚本，没有建制化单元测试和集成测试。要改进的话优先级：_assert_readonly_sql 绕过测试、SQL 参数注入测试、Agent mock 测试。

### Q11：最优先改进什么？

> 第一补测试，第二全局单例干掉（已做部分：LLM 和 Graph 改成了可注入模式），第三记忆系统从文件迁到数据库。

## 五、设计决策与取舍

### Q12：为什么 Qdrant 不选 Pinecone/Weaviate？

> 个人项目限制。Qdrant 最轻量，docker compose 一行配置。生产环境会根据数据量和运维成本重新评估。

### Q13：为什么 DeepSeek 不是 GPT？

> 成本。DeepSeek 推理成本约 GPT-4 的 1/10，中文效果无明显差距。

## 七、行为面试

### "最大的技术挑战？"

> SQL 自动修正。一开始想简单——报错就重试。后来发现很多错误是语义问题——LLM 用了错误的口径定义。需要把完整业务口径上下文传回修正节点，让 LLM 知道"为什么错了"而不是"语法不对"。

### "如果重做会怎么改？"

> 先搭测试架构再写代码。另外先做数据库设计——记忆系统用文件存储导致迁移成本很高。

---

## 八、面试中可主动展示的加分点

1. SQL 注入两种防御：select 参数白名单 + AST 全树遍历黑名单
2. BM25 分数归一化：sigmoid(score/5) 映射到 [0,1]
3. 父块-子块存储：检索用子块提高精度，回答用父块提供完整上下文
4. 熔断机制：Qdrant 和 ES 两路各自独立熔断计数器
5. 语义缓存混合策略：精确缓存 + 语义缓存
